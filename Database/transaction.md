# 트랜잭션(transaction)

## 🔹 트랜잭션(transaction)

**트랜잭션(transaction)** 이란 DBMS 또는 유사한 시스템에서 하나의 논리적인 작업 단위를 의미합니다. 하나의 트랜잭션은 여러 작업으로 구성될 수 있으며, 모두 성공하거나 또는 모두 실패해야 합니다.


### 1. 왜 트랜잭션이 필요할까?

예를 들어, 누군가에게 송금을 한다고 생각해보겠습니다. <br />
보통 송금은 아래 두 가지 작업으로 구성됩니다.

1. 내 계좌에서 출금
2. 상대방 계좌로 입금

그런데 만약 출금은 성공했는데, 입금이 실패했다면? <br />
돈은 빠져나갔지만, 상대방은 받지 못한 상황이며, **데이터의 일관성**이 깨진 상태입니다.

이런 문제를 막기 위해 입금과 출금을 **하나의 트랜잭션**으로 묶어야 합니다. <br />
그리고 중간에 어떤 문제가 생기면, **롤백(rollback)** 을 통해 모든 작업을 **이전 상태로 되돌려야** 합니다.

> 트랜잭션은 이렇게 데이터의 신뢰성과 일관성을 보장해주는 중요한 장치입니다.


### 2. 트랜잭션은 어떻게 동작할까?

트랜잭션은 데이터베이스에서 제공하는 `commit` 과 `rollback` 명령어로 제어됩니다. <br />
트랜잭션이 실행되면, 내부에서 수행된 모든 데이터 변경 작업은 먼저 **임시 상태**로 처리되며, 해당 트랜잭션을 실행한 세션에서만 보이고 다른 세션에서는 보이지 않습니다.

- `commit`: 모든 작업이 정상적으로 완료되면, 변경사항을 데이터베이스에 영구 반영합니다.
- `rollback`: 중간에 오류 발생 시, 트랜잭션 내에서 실행한 모든 작업을 이전 상태로 되돌립니다.

```sql
-- 트랜잭션 시작
START TRANSACTION;

-- 출금 처리: A 계좌에서 1000원 출금
UPDATE 계좌 SET 잔액 = 잔액 - 1000 WHERE 계좌번호 = 'A';

-- 입금 처리: B 계좌에 1000원 입금
UPDATE 계좌 SET 잔액 = 잔액 + 1000 WHERE 계좌번호 = 'B';

-- 만약 모든 작업이 정상적으로 완료되었다면 커밋
COMMIT;

-- 만약 오류가 발생했다면 롤백
ROLLBACK;
```

> 트랜잭션을 이용하면 데이터의 신뢰성과 일관성을 안전하게 유지할 수 있습니다.


## 🔹 트랜잭션의 특징 - ACID

트랜잭션이 데이터의 **정확성**과 **신뢰성**을 보장할 수 있는 이유는 바로 **ACID**라는 네 가지 특성을 따르기 때문입니다.

### 1. Atomicity (원자성)

**트랜잭션 내의 모든 작업은 전부 반영되거나, 전혀 반영되지 않아야 합니다.** <br />
즉, 트랜잭션은 하나의 **논리적인 작업 단위**로 묶이며, 중간에 일부만 처리되는 일은 없어야 합니다.

*ex. A 계좌에서 출금 후 B 계좌에 입금하는 도중에 중단되면 안 됩니다.*

이처럼 트랜잭션은 **전부 성공** 또는 **전부 실패**해야 하며, 이는 시스템의 **예측 가능성**과 **안정성**을 높이는 핵심입니다.

### 2. Consistency (일관성)

**트랜잭션이 실행되기 전과 실행된 후의 데이터는 항상 정합성(일관성)을 유지해야 합니다.** <br />
트랜잭션 중간에 데이터가 일시적으로 변하더라도, 최종 결과는 항상 **정의된 규칙을 만족하는 상태**여야 합니다.

*ex. 계좌 이체가 끝난 후엔 전체 계좌의 총 잔액이 바뀌지 않아야 합니다.*

일관성이 깨지면 시스템의 데이터 신뢰도에 큰 문제가 생기기 때문에, 트랜잭션은 항상 **유효한 상태로 데이터를 유지**하도록 보장됩니다.

### 3. Isolation (격리성)

**여러 트랜잭션이 동시에 실행되더라도 서로 간섭해서는 안 됩니다.** <br />
한 트랜잭션이 수행 중일 때, 다른 트랜잭션은 그 **결과를 볼 수도, 접근할 수도 없습니다.**

*ex. 내가 상품을 결제하는 동안 다른 사용자가 그 상품의 재고를 건드리면 안 됩니다.*

트랜잭션들은 독립적으로 실행되어야 하며, 실행 순서에 따라 결과가 달라지지 않아야 합니다. 이것이 격리성의 핵심입니다.

### 4. Durability (지속성)

**트랜잭션이 성공적으로 커밋되면, 그 결과는 시스템 장애가 발생해도 유지되어야 합니다.** <br />
즉, 일단 커밋된 데이터는 **절대 사라지지 않도록 보장**되어야 합니다.

*ex. 계좌 이체 완료 후 시스템이 다운되더라도, 데이터는 날아가면 안 됩니다.*

이를 위해 데이터베이스는 로그나 디스크 같은 **영구 저장 장치**에 트랜잭션 결과를 안전하게 기록합니다.

> ACID 속성이 보장되기 때문에, 트랜잭션을 통해 우리는 데이터의 정확성, 안정성, 신뢰성을 유지할 수 있습니다.
